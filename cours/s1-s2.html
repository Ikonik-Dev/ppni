🔐 Séquence 1.1 — Comprendre les menaces

Durée : 2h — 3 notions essentielles + atelier

💡 Idée clé : Une menace, c'est ce qui peut t'arriver. Un risque, c'est la probabilité que ça arrive. Une vulnérabilité, c'est la porte ouverte par où ça rentre. On apprend à fermer les portes.
10% — Théorie

Vocabulaire essentiel

  • Menace : virus, hacker, phishing, ransomware
  • Risque : probabilité + impact d'une attaque
  • Vulnérabilité : mot de passe faible, Wi-Fi public, logiciel non mis à jour
  • Vecteur d'attaque : e-mail, lien, clé USB, faux site
⏱ 20 min
70% — Pratique

Atelier : le mot de passe parfait

  • Tester son mot de passe sur howsecureismypassword.net
  • Créer une phrase de passe forte (méthode des 4 mots aléatoires)
  • Installer et tester un gestionnaire de mots de passe (Bitwarden)
  • Activer le MFA sur son compte Google ou GitHub
⏱ 55 min
20% — Collaboration

Discussion collective : vos habitudes

  • Combien de mots de passe différents utilisez-vous ?
  • Avez-vous déjà reçu un e-mail suspect ?
  • Partage d'expériences (sans honte — tout le monde apprend)
⏱ 25 min

🎣 Séquence 1.2 — Détecter le phishing

Durée : 1h40 — Cas réels + quiz interactif

70% — Pratique

Quiz phishing — vrai ou faux ?

  • 10 e-mails présentés : repérer les indices suspects
  • Indices à chercher : expéditeur, lien, ton urgent, fautes, logo
  • Correction collective avec explications
  • Cas réel : faux e-mail "Votre compte sera supprimé"
⏱ 50 min
20% — Collaboration

En binôme : analyser un e-mail

  • Chaque binôme reçoit 2 e-mails (1 vrai, 1 phishing)
  • Lister 3 preuves qui permettent de distinguer les deux
  • Présentation de 2 min par binôme
⏱ 30 min
10% — Théorie

Réflexes en cas d'attaque

  • Ne jamais cliquer sans vérifier l'URL
  • Signaler à l'ANSSI (cybermalveillance.gouv.fr)
  • Que faire si on a cliqué : démarche en 5 étapes

🐙 Séquence 1.3 — Découverte de GitHub

Durée : 2h — Guidé pas à pas

💡 GitHub c'est quoi ? Imagine un Google Drive, mais pour du code. Tu peux sauvegarder tes fichiers, suivre toutes les modifications, travailler à plusieurs, et montrer ton travail au monde entier. C'est la vitrine de tout développeur ou technicien numérique.
10% — Théorie

Vocabulaire Git/GitHub

  • Repository (repo) : ton dossier de projet
  • Commit : une sauvegarde commentée
  • Push : envoyer ses commits sur GitHub
  • README.md : la page d'accueil du projet
  • Local / Remote : ton ordi / le serveur GitHub
⏱ 20 min
70% — Pratique

Créer son compte GitHub

  • Aller sur github.com → créer un compte (e-mail pro ou perso)
  • Activer le MFA sur son compte GitHub
  • Personnaliser son profil (photo, bio courte)
  • Explorer l'interface : repos, explore, profil public
  • Installer GitHub Desktop (outil graphique pour débutants)
⏱ 60 min
20% — Collaboration

Trouver le compte des autres

  • Chaque stagiaire partage son URL GitHub dans le chat de groupe
  • Suivre le compte du formateur (Follow)
  • Explorer 2 repos publics de développeurs connus

🔁 Séquence 2.1 — Le flux Git de A à Z

Durée : 1h30

10% — Théorie

Schéma du flux Git

  • Espace de travail (ton ordi) → git add → Staging area
  • Staging area → git commit → Repo local
  • Repo local → git push → GitHub (remote)
  • GitHub → git pull → Repo local (récupérer les mises à jour)
⏱ 20 min
70% — Pratique

Atelier flux complet avec GitHub Desktop

  • Cloner son propre repo ppni-cybersecurite en local
  • Modifier le README dans un éditeur (VS Code ou Notepad)
  • Observer le changement dans GitHub Desktop (diff)
  • Écrire un message de commit clair et committer
  • Pusher et vérifier sur github.com que le changement apparaît
⏱ 55 min
20% — Collaboration

Revue des repos de la semaine dernière

10h30 – 12h30 Branches Git : travailler sans risque

🌿 Séquence 2.2 — Les branches Git

Durée : 2h

💡 Pourquoi des branches ? Imagine que tu travailles sur un document Word important. Tu veux tester une nouvelle version sans risquer de perdre l'original. Une branche, c'est exactement ça : une copie parallèle où tu expérimentes, et tu fusionnes quand c'est prêt.
70% — Pratique

Créer et fusionner une branche

  • Créer une branche amelioration-readme depuis GitHub Desktop
  • Ajouter une nouvelle section dans le README sur cette branche
  • Committer et pusher la branche
  • Ouvrir une Pull Request sur GitHub.com
  • Merger la PR dans main
  • Vérifier que main est bien à jour
⏱ 70 min
20% — Collaboration

Code review croisée

  • Chaque stagiaire laisse un commentaire sur la PR d'un camarade
  • Le formateur montre comment lire une diff
  • Discussion : qu'est-ce qu'un bon message de commit ?
⏱ 25 min
10% — Théorie

Bonnes pratiques Git

  • Messages de commit : verbe à l'infinitif, clair et court
  • 1 commit = 1 action précise (pas "modifications diverses")
  • Ne jamais forcer un push sur main sans revue
⏱ 25 min

15h00 – 17h00 Sécurité GitHub + bonnes pratiques de dépôt

🔒 Séquence 2.4 — GitHub en toute sécurité

Durée : 2h

70% — Pratique

Sécuriser son compte GitHub

  • Vérifier les paramètres de sécurité du compte GitHub
  • Comprendre les repos privés vs publics (cas d'usage)
  • Ne jamais committer de mot de passe ou clé API (démo de l'erreur à éviter)
  • Ajouter un fichier .gitignore basique à son repo
  • Configurer les notifications e-mail GitHub
⏱ 70 min
20% — Collaboration

Scénario de groupe

  • Cas pratique : "Un stagiaire a commité son mot de passe par erreur — que faire ?"
  • Discussion collective : retrouver la solution (git revert, contacter GitHub)
⏱ 20 min
10% — Théorie

Les secrets ne vont pas sur GitHub

  • Règle absolue : jamais de mot de passe, token API, clé SSH en clair dans un commit
  • Utiliser des variables d'environnement (.env) + .gitignore
  • GitHub Secrets pour les projets avancés (mention)
⏱ 10 min
